# ContentPlan

## Purpose

This document defines the content plan for the Alto Multimodal Trail website. It complements:

- `SiteDesign.md` — design intent, personas, IA, UX guidance
- `ImplementationPlan.md` — phased implementation and delivery plan

This file is intended for source control and ongoing editorial management. It tracks:

- page purpose
- primary and secondary personas
- key messages
- required content blocks
- calls to action
- evidence/document needs
- ownership/status fields to support execution

---

## Content Principles

The site content should follow these principles:

1. **Lead with clarity**
   - Use plain language.
   - Explain the proposal in concrete terms before making arguments for it.

2. **Organize by user questions, not document type**
   - Content should answer:
     - What is this?
     - Why does it matter?
     - What are the concerns?
     - What does the evidence show?
     - What is happening now?
     - How can I help?

3. **Be advocacy-oriented, but credible**
   - Use persuasive framing without hype.
   - Distinguish clearly between:
     - established facts
     - evidence-backed interpretation
     - open questions
     - advocacy positions

4. **Acknowledge concerns directly**
   - Concerns should be addressed fairly and clearly.
   - The site should not appear dismissive or one-sided.

5. **Use visuals in context**
   - Maps, figures, and diagrams should be embedded on the pages where they help users understand the topic.
   - Do not create top-level navigation based on content type.

6. **Support scanning**
   - Use strong headings, short introductory summaries, bullets where useful, and clear links to next steps.

7. **Always date time-sensitive content**
   - Updates, statements, and evidence summaries should show clear dates.

## Global Content Requirements

These elements should be applied consistently across the site.

### Page header requirements
Each page should have:

- clear page title
- short summary / standfirst
- breadcrumb on interior pages
- clear “next step” or related links

### Evidence and sourcing requirements
Where factual claims are made, track:

- source needed
- source type
- whether citation exists
- whether claim has been verified

### Visual/content asset tracking
Track whether each page needs:

- map
- diagram
- photo
- infographic
- downloadable PDF
- template letter
- quote/testimonial
- timeline item

# Recommended Editorial Workflow

## For each page
Track:
- Owner
- Status
- Draft link / source file
- Source needs
- Assets needed
- Reviewer
- Approval date
- Publish date

## Suggested workflow states
- Not Started
- In Progress
- Ready for Review
- Approved
- Published

## Review checklist
Before moving a page to `Approved`, confirm:
- page purpose is clear
- primary persona is served
- factual claims are sourced
- language is plain and scannable
- CTA is present
- visuals are in context
- page links to the next relevant step
- date is present if content is time-sensitive

---

# Open Design/Content Questions to Track

These should be tracked explicitly as unresolved until decided.

1. What is the final campaign/site name?
2. What is the precise wording of the proposal statement?
3. What is the current primary call to action?
4. Will the site collect email signups?
5. Will the site use analytics?
6. What is the preferred source/citation style?
7. What downloadable artifacts exist at launch?
8. What update cadence is realistic?
9. Who approves public-facing copy?
10. What visuals can be produced for launch?
